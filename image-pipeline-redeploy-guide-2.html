How This Project Differs from LinkForge — Read First
Why read this section?

If you recently deployed LinkForge, your muscle memory will trip you up here. This project uses a completely different architecture pattern. The table below shows where your assumptions will be wrong.

LinkForge Pro

CloudFormation creates everything in one command. HTTP API (v2). Lambda wired to API via integrations. Frontend points to API Gateway URL for all operations.

Image Optimization Pipeline

No CloudFormation — everything created manually with CLI. REST API (v1). Lambda triggered by S3 events, not API Gateway. Frontend uploads DIRECTLY to S3 (bypasses API Gateway entirely for the actual file).

Phase 1 — Create 3 S3 Buckets
Why S3 first? The IAM Role (Phase 2) references bucket ARNs in its permissions. Lambda functions (Phase 3) reference bucket names in their environment variables. The S3 event trigger (Phase 4) is attached to the input bucket. Everything downstream depends on the buckets existing first.
⚠️
Global rule for this entire guide — single quotes vs double quotes in zsh/bash:
Many commands below pass JSON to the AWS CLI via --policy, --policy-document, --cors-configuration, etc. The quoting rule is:

• No variables inside JSON → single quotes are fine: '{"key": "value"}'
• Variables inside JSON → must use double quotes + escape inner quotes: "{\"key\": \"${MY_VAR}\"}"

Single quotes in zsh prevent ALL variable expansion — '$BUCKET' passes the literal string $BUCKET to AWS, not your bucket name. This causes MalformedPolicy errors on IAM and S3 policy commands, and InvalidParameter errors on notification configs. Every JSON block in this guide that contains a $VARIABLE reference uses double quotes.

Why 3 separate buckets instead of one?

Input bucket: Needs PUT permission for pre-signed uploads and fires S3 events. Mixing it with processed images would cause the event trigger to fire on its own outputs — an infinite processing loop.

Output bucket: Needs GET permission for pre-signed downloads. Organized with resolution subfolders (1080p/, 720p/, 480p/) so the get-processed-images Lambda can construct predictable keys.

Frontend bucket: Needs static website hosting enabled with public read. Mixing static site files with image data would be confusing and could accidentally expose images.

aws s3 website s3://$FRONTEND_BUCKET/ \

What does website hosting do?

Normally S3 objects are accessed via the REST API endpoint (e.g. s3.amazonaws.com/bucket/key). With website hosting enabled, S3 creates a separate website endpoint (e.g. bucket.s3-website.ap-south-1.amazonaws.com) that behaves like a web server — it serves index.html for the root URL, handles 404s with the error document, and serves files with correct content-type headers. This is required for a browser to load the app correctly.

Issue 1 Hit: zsh: unknown file attribute: 3 
zsh: unknown file attribute: 3
Why it happened: zsh interpreted the number in the --public-access-block-configuration flag value (BlockPublicAcls=false,...) differently when pasted with the backslash continuation on the same line. This error is harmless — it means zsh got confused about the line break, but the AWS command itself still executed successfully. No AWS error = Step 3a completed fine. You can verify with: 
aws s3api get-public-access-block --bucket $FRONTEND_BUCKET --region ap-south-1
All four values should return false. The fix in the guide above puts the configuration value on its own indented line without the trailing backslash issue.

Issue 2 Hit: MalformedPolicy — Policy has invalid resource 
An error occurred (MalformedPolicy) when calling the PutBucketPolicy operation:
Policy has invalid resource
Why it happened: The bucket policy JSON was wrapped in single quotes ('...'). In zsh (and bash), single quotes prevent ALL variable expansion — the string $FRONTEND_BUCKET was passed literally to AWS instead of being expanded to image-pipeline-frontend-anshuman. AWS received the ARN as arn:aws:s3:::$FRONTEND_BUCKET/* which is not a valid ARN, so it rejected the policy.

The rule — single quotes vs double quotes in shell:
• '$VAR' → literal string $VAR (no expansion) — use for static JSON with no variables
• "$VAR" or "${VAR}" → expands to the variable's value — required whenever your JSON contains shell variables

Fix — use double quotes and escape inner quotes with backslashes: 
aws s3api put-bucket-policy \
  --bucket $FRONTEND_BUCKET \
  --region ap-south-1 \
  --policy "{
    \"Version\": \"2012-10-17\",
    \"Statement\": [{
      \"Sid\": \"PublicReadGetObject\",
      \"Effect\": \"Allow\",
      \"Principal\": \"*\",
      \"Action\": \"s3:GetObject\",
      \"Resource\": \"arn:aws:s3:::${FRONTEND_BUCKET}/*\"
    }]
  }"
Success response: No output at all — an empty response from put-bucket-policy means success. AWS only returns output on failure.

Remember for interviews: This single-quote vs double-quote issue will appear in EVERY step that passes JSON with shell variables to the AWS CLI — the IAM policy (Phase 2), the S3 event notification (Phase 4), and the API Gateway CORS integration responses (Phase 5). Always use double quotes + escaped inner quotes when your JSON contains $VARIABLE references.
Why two steps for public access?

AWS introduced "Block Public Access" settings in 2018 as a safety guard — even if you add a bucket policy, the block settings override it and prevent public access. You must explicitly disable the block settings first, THEN add the bucket policy. Doing only one of these is a common mistake that results in 403 errors when trying to load the frontend website.

The bucket policy uses Principal: "*" (anyone) and Action: s3:GetObject (read-only) on all objects (/*). This allows any browser to fetch HTML/CSS/JS files — which is exactly what a public website needs.

How to verify it worked: aws s3api get-bucket-policy --bucket $FRONTEND_BUCKET --region ap-south-1 — the returned policy should show your bucket name expanded correctly (not $FRONTEND_BUCKET literally).

Enable CORS on the input bucket (browsers need this for direct PUT)
# Use double quotes so $INPUT_BUCKET expands correctly
aws s3api put-bucket-cors \
  --bucket $INPUT_BUCKET \
  --region ap-south-1 \
  --cors-configuration '{
    "CORSRules": [{
      "AllowedOrigins": ["*"],
      "AllowedMethods": ["PUT", "POST", "GET"],
      "AllowedHeaders": ["*"],
      "MaxAgeSeconds": 3000
    }]
  }'
✅
Single quotes are safe here — the CORS JSON contains no shell variables, so single quotes work fine. The bucket name is passed via --bucket $INPUT_BUCKET (outside the quotes) which does expand correctly. Single quotes are only a problem when variables appear INSIDE the quoted JSON string.
Why does the input bucket need CORS?

The upload flow works like this: browser calls API Gateway → gets a pre-signed S3 URL → browser then PUTs the file directly to S3 using that URL. This direct S3 PUT comes from the user's browser (a different origin than S3). Browsers enforce CORS — they send a preflight OPTIONS request before every cross-origin PUT. Without CORS configured on the S3 bucket, S3 returns no CORS headers and the browser blocks the upload before it even starts. The actual file never reaches S3.

Enable CORS on the output bucket (browsers need this for direct GET)
aws s3api put-bucket-cors \
  --bucket $OUTPUT_BUCKET \
  --region ap-south-1 \
  --cors-configuration '{
    "CORSRules": [{
      "AllowedOrigins": ["*"],
      "AllowedMethods": ["GET"],
      "AllowedHeaders": ["*"],
      "MaxAgeSeconds": 3000
    }]

Why does the output bucket also need CORS?

Same reason — when the user clicks "Download", the browser follows the pre-signed URL directly to S3 to fetch the processed image. This is a cross-origin GET from the user's browser. Without CORS on the output bucket, the download button works in some browsers (those that don't enforce CORS on GETs) but fails in others with a CORS error.

Phase 2 — Create IAM Role for Lambda
Why before Lambda? When you run aws lambda create-function you must supply a --role ARN at creation time. The role must exist before the function is created. All 3 Lambda functions will share this single role.
1
Create the IAM role with Lambda trust policy
aws iam create-role \
  --role-name Lambda-Image-Processing-Role \
  --assume-role-policy-document '{
    "Version": "2012-10-17",
    "Statement": [{
      "Effect": "Allow",
      "Principal": {"Service": "lambda.amazonaws.com"},
      "Action": "sts:AssumeRole"
    }]
  }'
What is the trust policy and why does it matter?

An IAM Role by itself grants permissions — but you also need to specify WHO is allowed to use (assume) the role. This "trust policy" is a separate document that says "the Lambda service (lambda.amazonaws.com) is allowed to assume this role." Without it, Lambda cannot pick up the role's permissions when your function runs. This is a frequent point of confusion: permissions policy = what the role CAN do; trust policy = WHO can use the role.

🔴
Issue Hit: EntityAlreadyExists — Role already exists 
An error occurred (EntityAlreadyExists) when calling the CreateRole operation:
Role with name Lambda-Image-Processing-Role already exists.
Why it happened: The IAM role was created during a previous deployment and still exists in your AWS account. IAM roles are global (not region-specific) and persist until explicitly deleted. Unlike S3 buckets where you needed a new suffix, IAM roles with the same name across deployments is fine — you can simply reuse the existing one.

Fix — skip create-role and just fetch the existing ARN: 
ROLE_ARN=$(aws iam get-role --role-name Lambda-Image-Processing-Role \
  --query 'Role.Arn' --output text)
echo $ROLE_ARN
Then continue with Steps 2 and 3 (attaching policies). The attach-role-policy and put-role-policy commands are safe to re-run on an existing role — they are idempotent (attaching an already-attached policy just returns success silently).

Remember for re-deployments: IAM roles, policies, CloudWatch log groups, and Lambda functions all persist between deployments. Only S3 objects are truly ephemeral. The pattern is: try to create, if EntityAlreadyExists get the existing resource ARN and continue.
2
Attach CloudWatch Logs permission (built-in AWS policy)
aws iam attach-role-policy \
  --role-name Lambda-Image-Processing-Role \
  --policy-arn arn:aws:iam::aws:policy/service-role/AWSLambdaBasicExecutionRole
What does AWSLambdaBasicExecutionRole grant?

This AWS-managed policy grants permission to create CloudWatch log groups, log streams, and write log events. Without it, your Lambda functions run but you cannot see any print() output or error tracebacks in CloudWatch Logs — debugging becomes impossible. This is the minimum policy every Lambda function needs.

3
Attach S3 read/write permissions (custom inline policy)
# IMPORTANT: Use double quotes + escaped inner quotes so $INPUT_BUCKET and $OUTPUT_BUCKET expand
aws iam put-role-policy \
  --role-name Lambda-Image-Processing-Role \
  --policy-name S3ImagePipelineAccess \
  --policy-document "{
    \"Version\": \"2012-10-17\",
    \"Statement\": [
      {
        \"Effect\": \"Allow\",
        \"Action\": [\"s3:GetObject\", \"s3:PutObject\", \"s3:HeadObject\"],
        \"Resource\": [
          \"arn:aws:s3:::${INPUT_BUCKET}/*\",
          \"arn:aws:s3:::${OUTPUT_BUCKET}/*\"
        ]
      },
      {
        \"Effect\": \"Allow\",
        \"Action\": \"s3:GeneratePresignedUrl\",
        \"Resource\": \"*\"
      }
    ]
  }"
⚠️
Same single-quote vs double-quote rule applies here: ${INPUT_BUCKET} and ${OUTPUT_BUCKET} appear inside the JSON string. Use double quotes on the outer string and escape all inner JSON quotes with backslashes. If you use single quotes, the Resource ARNs will contain the literal text $INPUT_BUCKET and the policy will silently apply to a bucket that does not exist.
Breaking down the S3 permissions needed

s3:GetObject on input bucket: Lambda 2 (ImageProcessor) downloads the original image from the input bucket to /tmp for processing.

s3:PutObject on output bucket: Lambda 2 uploads 3 processed images to the output bucket after resizing/compressing.

s3:HeadObject: Lambda 3 (get-processed-images) may check if the processed files exist before generating download URLs.

s3:GeneratePresignedUrl on Resource: *: Pre-signed URL generation is a client-side SDK operation — it uses your IAM credentials to sign a URL that can be used by anyone. It technically doesn't need a specific resource because no S3 API call is made — the SDK signs the URL locally using your credentials. However the effective S3 operation (PUT/GET) is still bound by the bucket policies above.

Why after IAM? Lambda create-function requires the role ARN at creation time. Why before API Gateway (Phase 5)? API Gateway integrations need the Lambda function ARNs. Why before S3 event trigger (Phase 4)? The S3 trigger notification points to a specific Lambda ARN — the function must exist first.

Critical difference from LinkForge: Lambda 2 (ImageProcessor) uses Pillow which is NOT available in the Lambda runtime. You MUST bundle Pillow into the ZIP. Lambda 1 and 3 have no external dependencies — only boto3, which IS pre-installed. Two different packaging approaches for this one project.
1

What this function does and why it's simple to package

This function receives {"filename": "photo.jpg", "contentType": "image/jpeg"}, calls s3.generate_presigned_url("put_object", ...), and returns the signed URL. It only uses boto3 which is pre-installed in every Lambda Python runtime — no pip install needed, the .py file alone is the entire deployment package.

--timeout 10: Pre-signed URL generation is instant (no network call — it's a local crypto signing operation). 10 seconds is generous but appropriate for a URL generation endpoint.

--memory-size 128: Minimum memory. URL generation is trivial computation — no reason to pay for more.

3
Package Lambda 2 — ImageProcessor (Pillow must be bundled)
# Create a clean packaging directory
mkdir -p build/image-processor-pkg

# Install Pillow into that directory (this is the critical step)
pip3 install pillow --target build/image-processor-pkg --platform manylinux2014_x86_64 \
  --implementation cp --python-version 3.11 --only-binary=:all:

# Copy the Lambda code into the same directory
cp lambda-functions/image-processor/lambda_function.py build/image-processor-pkg/

# Zip everything together — Pillow + lambda_function.py at the root of the ZIP
cd build/image-processor-pkg
zip -r ../image_processor.zip .
cd ../..
PartWhat it does and why exactly this way
pip3 install pillow --targetInstalls Pillow into a specific directory instead of the system Python. --target means "put the package files here" — this is the standard way to create Lambda deployment packages with dependencies.
--platform manylinux2014_x86_64CRITICAL: Pillow is a C extension library. Your Mac/Windows has a different binary format than the Lambda Linux runtime. This flag forces pip to download the Linux-compatible binary wheel instead of the binary for your local machine. Without this, Pillow will import successfully on your Mac but crash with ImportError on Lambda.
--implementation cp --python-version 3.11Specifies the exact CPython 3.11 binary to download — must match the Lambda runtime (python3.11). This ensures the Pillow C extensions are compiled for the right Python ABI.
--only-binary=:all:Forces pip to only download pre-built wheel files (no compilation from source). Without this, pip might try to compile Pillow from source code, which requires build tools not available in the sandboxed environment.
zip -r ../image_processor.zip .Zips everything in the packaging directory — both lambda_function.py and the entire Pillow package directory tree — into a single ZIP. Lambda extracts this ZIP into the execution environment, making Pillow importable by your code.
🔴
Most common packaging mistake: missing the Linux platform flag 
# WRONG — installs Pillow for your local OS (Mac/Windows)
pip3 install pillow --target build/image-processor-pkg

# CORRECT — installs the Linux binary that Lambda can run
pip3 install pillow --target build/image-processor-pkg \
  --platform manylinux2014_x86_64 \
  --implementation cp --python-version 3.11 --only-binary=:all:

Why timeout 59 and memory 1024 — very different from Lambdas 1 and 3

Timeout 59 seconds: Lambda has a hard maximum of 15 minutes, but a processing function needs careful tuning. A 14MB image takes ~2.5 seconds to download, process at 3 resolutions, and upload 3 files. A 50MB image near the limit takes ~6 seconds. Setting 59 seconds gives ample buffer while keeping the function well within Lambda's free tier duration billing. Setting it to the Lambda maximum (900s) would be wasteful — if there's a bug causing an infinite loop, you'd pay for 15 minutes instead of failing fast at 59 seconds.

Memory 1024 MB: Pillow loads the entire image into memory for processing. A 14MB JPEG decompressed into raw pixels becomes approximately 100–200 MB in memory. Processing 3 resolutions simultaneously means peak memory usage can hit 300–400 MB. 1024 MB gives a safe ceiling. Important: Lambda's CPU allocation is directly proportional to memory — more memory = more CPU = faster processing. The image processor benefits from more CPU to run Pillow's compression algorithms faster.

Key insight — this Lambda is NEVER called by API Gateway

Notice there is no API Gateway route pointing to ImageProcessorLambda-Shivam. It is triggered exclusively by the S3 event configured in Phase 4. When a user uploads an image (via the pre-signed URL from Lambda 1), the image lands in the input bucket, and S3 automatically fires an event that invokes this Lambda. The user never knows this Lambda exists — they just wait ~5 seconds and then request download URLs from Lambda 3.

Phase 4 — S3 Event Trigger (Wires S3 to ImageProcessor)
Why Phase 4 after Lambda? The S3 event notification points to a specific Lambda ARN. The Lambda must exist before S3 can be configured to call it. Similarly, you must grant Lambda invoke permission before adding the notification — otherwise S3 cannot deliver the event.
💡
This is the "magic" that makes the pipeline automatic: Once this trigger is configured, every single image uploaded to the input bucket (regardless of how it got there — pre-signed URL, CLI, console) automatically triggers the ImageProcessor Lambda. No polling, no manual invocation. S3 pushes an event synchronously to Lambda.
1
Grant S3 permission to invoke the ImageProcessor Lambda
aws lambda add-permission \
  --function-name ImageProcessorLambda-Shivam \
  --statement-id s3-trigger-permission \
  --action lambda:InvokeFunction \
  --principal s3.amazonaws.com \
  --source-arn arn:aws:s3:::$INPUT_BUCKET \
  --source-account $ACCOUNT_ID \
  --region $REGION
FlagWhat it does and why
--principal s3.amazonaws.comGrants S3 (not API Gateway, not a user) permission to invoke this Lambda. S3 uses its own service principal — different from the apigateway.amazonaws.com used in LinkForge.
--source-arn arn:aws:s3:::$INPUT_BUCKETRestricts to ONLY the input bucket. Without this, any S3 bucket in your account (or even other accounts) could invoke the ImageProcessor — a security risk. Scoping to one bucket ARN ensures only uploads to this specific bucket trigger processing.
--source-account $ACCOUNT_IDAdditional security restriction — only events from your own account. Prevents "confused deputy" attacks where a malicious bucket in another account could trigger your Lambda.
Why must this permission be added BEFORE the S3 notification?

When you configure the S3 bucket notification in Step 2, AWS validates that the target Lambda function actually allows S3 to invoke it. If the permission doesn't exist, the put-bucket-notification-configuration command itself fails with "Unable to validate the following destination configurations." Always add the Lambda permission first, then configure the S3 notification.

2
Configure the S3 bucket notification
PROCESSOR_ARN="arn:aws:lambda:$REGION:$ACCOUNT_ID:function:ImageProcessorLambda-Shivam"

aws s3api put-bucket-notification-configuration \
  --bucket $INPUT_BUCKET \
  --region $REGION \
  --notification-configuration "{
    \"LambdaFunctionConfigurations\": [{
      \"LambdaFunctionArn\": \"$PROCESSOR_ARN\",
      \"Events\": [\"s3:ObjectCreated:*\"]
    }]
  }"
What does s3:ObjectCreated:* mean?

s3:ObjectCreated:* is a wildcard that matches any object creation event — this includes s3:ObjectCreated:Put (direct PUT), s3:ObjectCreated:Post (multipart upload), s3:ObjectCreated:Copy (object copy), and s3:ObjectCreated:CompleteMultipartUpload. Using * ensures the ImageProcessor fires regardless of how the image arrived in the bucket. If you used Put only, a large image uploaded via multipart would not trigger processing.

The event payload S3 sends to Lambda includes the bucket name, object key, object size, and ETag. The ImageProcessor Lambda reads these from event['Records'][0]['s3'].

Why does REST API need manual CORS setup with OPTIONS (unlike HTTP API v2)?

HTTP API (v2) has a CorsConfiguration block that handles everything automatically. REST API (v1) has no such block — you must manually create an OPTIONS method on every resource and configure it with a MOCK integration (a fake endpoint that returns CORS headers without calling Lambda). The MOCK integration returns a 200 with the three CORS headers browsers need: Allow-Origin, Allow-Methods, Allow-Headers. This is the "preflight response" that browsers check before sending the actual POST.

The Lambda functions also return Access-Control-Allow-Origin: * in their response headers — this handles the "actual response" CORS check. You need CORS on both the OPTIONS preflight AND the actual response.

Why does REST API (v1) need an explicit deployment step?

REST API in API Gateway uses a "deployment + stage" model. A deployment is a snapshot of your API configuration at a point in time. A stage (prod, dev, staging) points to a specific deployment. This design means you can have different stages point to different versions of your API — useful for blue/green deployments. The downside: every time you change a method, integration, or CORS setting, you must create a new deployment and it takes effect in the stage only after the deployment. This is different from HTTP API v2 which has AutoDeploy — changes take effect immediately.

If you make any change to the API after this step, you MUST run create-deployment again. Forgetting this is the most common source of confusion in API Gateway v1 — "I changed the Lambda but the API still behaves the same."

The original generate-upload-url and get-processed-images code had the bucket names as string literals — os.environ was never called. The env var was set but completely ignored by the code.

The two-step fix:
1. Update the source code to read from env vars (already done in the repo by Kiro's edit during this session)
2. Re-zip and re-deploy the Lambda functions: 
cd lambda-functions/generate-upload-url
zip ../../build/generate_upload_url.zip lambda_function.py && cd ../..
aws lambda update-function-code \
  --function-name generate-upload-url \
  --zip-file fileb://build/generate_upload_url.zip \
  --region ap-south-1

cd lambda-functions/get-processed-images
zip ../../build/get_processed_images.zip lambda_function.py && cd ../..
aws lambda update-function-code \
  --function-name get-processed-images \
  --zip-file fileb://build/get_processed_images.zip \
  --region ap-south-1
Verify the fix worked: Run the curl test again — the uploadUrl should now say image-pipeline-input-anshuman.s3.amazonaws.com.

Lesson: setting env vars on a Lambda that doesn't read them does nothing. Always check whether the function code actually calls os.environ.get("VAR_NAME"). If it has hardcoded values, the code must be updated and redeployed.

PNG Transparency Bug — What It Is and How It's Fixed

🔴
Bug: "cannot write mode RGBA as JPEG" — affects all PNG files

PNG images can be saved in RGBA mode (Red, Green, Blue, Alpha). JPEG format does not support an alpha channel. When the ImageProcessor tried to save an RGBA image as JPEG, Pillow threw a fatal error and the entire processing failed — no output files were created.

Fix already in lambda_function.py (lines 29-36): 
# After resize, before saving — convert mode if needed
if resized_image.mode == 'RGBA':
    rgb_image = Image.new('RGB', resized_image.size, (255, 255, 255))  # white bg
    rgb_image.paste(resized_image, mask=resized_image.split()[3])      # alpha as mask
    resized_image = rgb_image
elif resized_image.mode != 'RGB':
    resized_image = resized_image.convert('RGB')  # grayscale, CMYK, etc.

Transparent areas become white. All image formats (JPEG, PNG, BMP, TIFF, GIF, WebP) now process successfully. This fix is already in the repo — you do not need to add it. But you MUST know about it for an interview.

Talking Points for Interviewers

Why pre-signed URLs instead of uploading through API Gateway?

API Gateway has a 10 MB request body limit. Large images would fail. Pre-signed URLs let the browser upload directly to S3 — no size limit, no API Gateway bottleneck, no bandwidth cost through API Gateway. The API just generates the credential, not the data.

Why S3 event trigger instead of the API calling Lambda?

Decoupling. The upload and processing are independent operations. Processing is asynchronous — the user doesn't wait. The same pipeline works regardless of how the image arrives (UI, CLI, another system). Serverless event-driven design: each service does one thing and reacts to events.

Why Pillow at quality 85 and not 100?

JPEG quality is perceptually logarithmic — 85 is visually near-identical to 100 but produces 70–90% smaller files. Quality 100 produces lossless-like files with minimal compression. 85 is the industry standard for web images — Netflix, YouTube, and most CDNs use similar settings.

Why 3 separate S3 buckets?

Separation of concerns + security. Input bucket fires S3 events — it would create an infinite loop if the ImageProcessor wrote back to it. Output bucket needs different CORS (GET only). Frontend needs public read. Mixing them violates least-privilege and creates debugging nightmares.

Why REST API v1 and not HTTP API v2?

REST API v1 was the original choice — it existed before v2 and offers more features (request validation, API keys, usage plans, resource policies). The project predates the v2 adoption. In a greenfield project today, HTTP API v2 would be the better choice for a simple proxy pattern like this — cheaper, simpler CORS, automatic deployment.

How does the PNG transparency fix work?

PNG supports RGBA (with alpha channel). JPEG does not support transparency. PIL raises an error if you try to save RGBA as JPEG. The fix: create a white RGB background canvas, paste the RGBA image using the alpha channel as a mask (transparent pixels become white), then save as JPEG. All formats now supported.

Why 1024 MB memory for ImageProcessor?

Two reasons: (1) A 14 MB JPEG decompressed into raw pixels = ~100-200 MB in RAM. Processing 3 resolutions risks OOM at 128 MB. (2) Lambda CPU is proportional to memory — more memory = more CPU = faster Pillow operations. 1024 MB struck the right balance of speed vs cost for the testing results we saw.

What would you add next?

WebP output (30% smaller than JPEG at same quality). CloudFront CDN in front of both S3 buckets for HTTPS and edge caching. SNS/SQS for async notification when processing completes (instead of the frontend polling and waiting). Custom resolution selection. Batch processing via S3 batch operations.
